## 1. 需求复述

### 1.1 痛点

CLI 工具链(inp-tool / wizard / REPL / sweep)在 Win7 用户面前存在 3 个具体障碍:

| 痛点 | 现状 | 影响 |
|---|---|---|
| **不熟悉 CLI** | Win7 工程师多从 CFD++ GUI(MFC,老)切入,命令行有心理门槛 | sweep YAML/JSON 写错字段报错不友好 |
| **改 .inp 找不到字段** | `mcfd.inp` 700+ 行,字段在 physics/guiopts/chemistry 等多个 block 里 | `set` 命令要查 `inp-tool parse` 输出 |
| **方程/湍流联动** | v0.9.1(预计)新增 2T 温度联动、SST k-ω 反算、species mass/mole 换算 | CLI 形式是 `turb I=0.01 L=0.01`,GUI 可点选 + 实时预览 |

### 1.2 终端用户画像

| 角色 | 占比(估) | 关键场景 |
|---|---|---|
| **CFD 工程师**(主要用户) | 60% | 改单 .inp 来流参数;批量生成攻角/马赫扫描;对比两版 .inp 差异 |
| **任务编排者**(集群用户) | 25% | sweep 整算例目录 + 配 pbs;查看 manifest |
| **项目经理 / 评审** | 15% | 看 .inp 改了什么(diff)、跑哪些 case、sweep 结果概览 |

### 1.3 关键场景(按优先级)

1. **单文件编辑**(最高频):打开 .inp → 树形浏览 blocks/stmts → 双击改值 → 保存
2. **Sweep 配置可视化**:YAML/JSON 表单化;实时预览笛卡尔积规模;支持 dry-run
3. **方程/湍流模型感知向导**:复用 v0.9.1 的 `detect_equations()` + 3 个 Preset,以向导形式展示
4. **Diff 对比**:双栏并排显示两个 .inp,标红差异
5. **Sweep 结果浏览**:列出生成的子目录,显示每个 case 的关键参数

### 1.4 非目标(明确不做)

- 取代 CFD++ 自己的 GUI(那是另一软件);本 GUI 是"前处理配置"工具
- 在线协作 / 多用户 / 云同步
- 3D 网格可视化(那是 ParaView / FieldView 的事)
- 求解器集成(不调 CFD++ solver)
- Win7 之外的旧系统(Win XP / Vista)—— Win7 SP1 是底线
- 移动端 / Web 端(走 FastAPI `[api]` 已有,Web 不是 v0.10 scope)

## 2. 技术选型对比

### 2.1 候选方案

| 方案 | Win7 支持 | Python 3.8 兼容 | PyInstaller 体积代价 | 外观 | 学习曲线 | 备注 |
|---|---|---|---|---|---|---|
| **Tkinter** (stdlib) | 原生(Win7 自带 Tcl/Tk 8.5+)| 内置 | 0 MB(已在 stdlib) | 朴素(Win7 风格)| 低 | widgets 较旧,ttk 可现代化 |
| **wxPython Phoenix 4.2+** | 原生 Win32 | 4.2.0 支持 Py3.8 | +20-30 MB(wxWidgets 库 + dll)| **原生外观**(Win7 = Windows 7 chrome)| 中 | Phoenix 是 wxWidgets 3.x 的 Python 绑定 |
| **PySide2 / PyQt5 5.15.x** | ✅ | PyQt5 5.15.0+ 兼容 Py3.8 | **+50-80 MB**(Qt5 dll + plugins) | 现代(支持 QSS 样式)| 中-高 | Qt 5.15 是最后支持 Win7 的 LTS 版(2020 末 EOL) |
| **PySimpleGUI** (Tkinter 包装)| (基于 Tkinter) | ✅ | +0-1 MB(纯 Python) | Tkinter 朴素 | 低 | 4.x 改了 license,商业注意 |
| **HTMLayout / cefpython** | ✅ | ✅ | +30-50 MB | 网页级(cef 嵌 Chromium) | 高 | cefpython 用 Chromium 70+,Win7 上边缘情况多 |

> **重要事实:Qt 5.15 是 Qt 公司承诺支持 Win7 的最后版本(2020 年 12 月 EOL)。** PyQt5 / PySide2 5.15.x 是 Win7 上能用 Qt 的天花板。Qt 6.x 明确不支持 Win7。

### 2.2 方案对比矩阵

| 维度 | Tkinter | wxPython | PyQt5/PySide2 | PySimpleGUI |
|---|---|---|---|---|
| **Win7 稳定性** | ★★★★★ | ★★★★★ | ★★★★(Qt 5.15 末班车)| ★★★★★ |
| **打包体积** | ★★★★★(0 MB)| ★★★(20-30 MB)| ★★(50-80 MB)| ★★★★★(0-1 MB)|
| **外观** | ★★ | ★★★★★(原生)| ★★★★(可定制)| ★★(基于 Tk)|
| **树形控件** | ttk.Treeview 够用 | wx.TreeCtrl 强大 | QTreeWidget 强大 | Tree element 简单 |
| **表单(自动布局)** | grid/place 手写 | sizers | layouts(可拖) | 1 行 1 控件(简单) |
| **打包 hook** | 内置无需 hook | PyInstaller 有现成 hook | PyInstaller 有 hook(但要小心) | 同 Tkinter |
| **中文 i18n** | 自由 | 自由 | tr() 完善 | 自由 |
| **社区/文档** | 中 | 中 | 强 | 中 |
| **Qt 风格"现代感"** | ❌ | ❌ | ✅ | ❌ |

### 2.3 推荐:**PySide2 5.15.x** (Qt for Python)

**核心理由:**

1. **控件丰富**:`QTreeWidget` 展示 .inp 块/语句层级,`QTableWidget` 改字段,`QFormLayout` 配 sweep / 检测报告,`QTextEdit` 预览 diff —— 比 Tkinter 手写 grid 省 60% 代码
2. **外观**:Qt 风格在 Win7 上仍现代(比 MFC 老 CFD++ GUI 强),可加 QSS 改色
3. **PyInstaller 兼容**:官方有 PyInstaller hook(`PyInstaller/hooks/hook-PySide2.QtCore.py` 等),成熟
4. **API 友好**:signal/slot 模型比 wxPython 事件驱动更适合 .inp 这种"改值→刷新预览"模式
5. **风险点明确**:Qt 5.15 EOL 是已知约束,文档/打包都成型;不是探索性技术
6. **Linux 同代码**:`inp-tool-gui` 在 Linux 跑得一样好,无需换框架

**为什么不选 wxPython / Tkinter:**

- **Tkinter** 太朴素,树形/表格要手写很多东西,改 .inp 的 700 行展示会卡(`ttk.Treeview` 1 万行实测会卡)
- **wxPython** 也是好选择(体积比 PySide2 小),但 API 偏 C++ 风格,signal/slot 不如 Qt 自然
- **PySimpleGUI** 适合简单表单,不适合 v0.9.1 的方程/湍流多步骤向导

**为什么不选 cefpython / HTMLayout:**

- 嵌浏览器要解决 Win7 上的字体/DPI 问题,反而比 Qt native 更折腾
- 本项目不需要 web 渲染能力

### 2.4 PySide2 vs PyQt5 抉择

| 维度 | PySide2 | PyQt5 |
|---|---|---|
| **License** | **LGPL**(商业友好)| GPL(商业要买 license) |
| **API** | 几乎一样 | 几乎一样 |
| **打包** | 一样 | 一样 |
| **官方维护** | Qt 公司 | Riverbank |

→ **选 PySide2**(LGPL + Qt 官方绑定),5.15.2.1 是最后稳定版(支持 Py3.8 + Win7)。

## 3. 推荐方案 + 理由

### 3.1 选型结论

| 项 | 选型 |
|---|---|
| **GUI 框架** | **PySide2 5.15.2.1**(Qt for Python,LGPL) |
| **打包工具** | PyInstaller 6.16.0(已用) |
| **入口** | `inp-tool-gui` console_script / `python -m inp_tool_gui` |
| **extras** | 新增 `[gui]` optional-dependency:`PySide2==5.15.2.1` |
| **核心原则** | GUI 是**并列包**于 CLI,wizard/REPL 不动,共享 `parser/writer/model/diff/sweep` 零依赖核心 |

### 3.2 与现有 CLI 的关系

```
                 ┌────────────────────┐
                 │   inp_tool core    │  ← 零依赖(纯 stdlib)
                 │  parser / writer / │
                 │  model / diff /    │
                 │  sweep / equations │
                 │  / wizard / repl   │
                 └────────────────────┘
                      ▲           ▲
                      │           │
            ┌─────────┘           └─────────┐
            │                              │
   ┌────────┴────────┐            ┌────────┴────────┐
   │  inp-tool CLI   │            │  inp-tool-gui   │  ← v0.10 新增
   │  (现有)         │            │  (PySide2)      │
   │  inp-tool-api   │            │                 │
   └─────────────────┘            └─────────────────┘
```

- **并列**关系,不是包装。CLI 继续存在(批处理 / 远程 / 服务器更合适)
- GUI 直接 `import inp_tool`,**复用** parser/writer/diff/sweep/equations(零依赖核心)
- wizard 4 步交互 → GUI 4 步对话框,逻辑层(`WizardSession` / `detect_equations()`)完全复用

### 3.3 extras 配置

`pyproject.toml` 新增:

```toml
[project.optional-dependencies]
gui = [
    "PySide2==5.15.2.1",   # Win7 兼容的 Qt 最后稳定版
]
build = [
    "pyinstaller==6.16.0",
]

    "pyinstaller==6.16.0",
    "PySide2==5.15.2.1",

```

**注意**:不把 PySide2 放到 `dependencies`(会污染零依赖核心)。装 GUI 才 `pip install ".[gui]"`。

**为什么独立包 `inp_tool_gui/` 而不是 `inp_tool/gui/`:**

1. **零依赖原则保持**:若放 `inp_tool/gui/`,`from inp_tool import parser` 会触发 `inp_tool/gui/__init__.py` 的 import 链(PySide2 是隐式 import),破坏"零依赖"承诺
2. **PyInstaller spec 干净**:`inp_tool_gui` 单独 spec(`inp_tool_gui.spec`),与 CLI spec 互不干扰
3. **可选安装边界清晰**:`pip install ".[gui]"` 只装 PySide2,不会污染只跑 CLI 的环境

### 4.3 GUI ↔ Core 边界

```python
# inp_tool_gui/controllers/file_controller.py
# 业务逻辑完全用 inp_tool core,GUI 不直接动 InpFile 的内部状态
from inp_tool.parser import parse_file
from inp_tool.writer import write as write_inp
from inp_tool.model import InpFile, Block, Stmt

class FileController:
    def __init__(self):
        self._inp: InpFile | None = None
        self._path: Path | None = None

    def open(self, path: str) -> InpFile:
        self._inp = parse_file(path)
        self._path = Path(path)
        return self._inp

    def save(self, path: str | None = None) -> None:
        out = path or str(self._path)
        write_inp(self._inp, out)

    def set_value(self, block_name: str, key: str, value, *, block_idx=0):
        b = self._inp.get_block(block_name, block_idx)
        if not b.set(key, value):
            b.append(key, value)  # 不存在就追加
```

**核心约束**:
- GUI 进程只 import `inp_tool`(零依赖),**不** import `fastapi`/`pydantic` 等 [api] 依赖
- GUI 可调用 `detect_equations` / `CaseSweep.generate` / `TurbulenceKOmegaPreset.apply` 等高层 API
- 业务逻辑(预校验、归一化、公式)在 core,GUI 只是"展示 + 触发"

### 4.4 状态管理

**主窗口状态:**

```python
class MainWindow(QMainWindow):
    def __init__(self):
        super().__init__()
        self.file_ctrl = FileController()     # 当前文件
        self.edit_ctrl = EditController()      # 改值/undo
        self.sweep_ctrl = SweepController()   # sweep 任务
        self.detect_ctrl = DetectController() # 方程检测
        self._undo_stack: list[UndoEntry] = []  # 改值历史
        self._is_dirty: bool = False           # 文件有未保存改动
```

**改值流程**(`QTreeWidget` 双击某 Stmt 的 value):

```
1. user_dblclick(stmt, value_idx)
   ↓
2. ValueEditorDialog(value) → 新值
   ↓
3. FileController.set_value(block, key, new_value)  ← 用 inp_tool core API
   ↓
4. InpFile 修改(Stmt.value.raw / typed)
   ↓
5. 推 undo: UndoEntry(block, key, old_value, new_value)
   ↓
6. main_window._is_dirty = True
   ↓
7. 触发 signal:inp_modified → tree_view.refresh_row()
   ↓
8. 标题栏加 *
```

### 4.5 文件 I/O

| 场景 | 入口 | 错误处理 |
|---|---|---|
| **打开 .inp** | `QFileDialog.getOpenFileName(filter="mcfd.inp (*.inp)")` | `parse_file` 抛 `ParseError` → `QMessageBox.critical` |
| **保存** | `Ctrl+S` / 菜单 | 路径不存在 → 询问新建;InpFile 校验失败 → 警告 + 让用户决定 |
| **另存为** | `Ctrl+Shift+S` | 同上 |
| **加载 sweep 配置** | `inp-tool-gui` 启动后 sweep 标签页 | YAML 错 → 显示行号(用 `yaml.YAMLError.problem_mark`) |
| **导出 manifest** | sweep 完成后"导出 manifest"按钮 | 路径写入失败 → 错误对话框 |

### 4.6 错误处理

- **分三层**:
  1. **Core 层**:`ParseError` / `SweepValidationError` / `TwoTemperatureError` 等(已有)
  2. **Controller 层**:捕获 + 转 `GuiError` dataclass`(code, message, details, suggestion)`
  3. **Widget 层**:统一 `error_dialog(self, e: GuiError)`,标题 / 详情 / 建议修复
- **不静默吞错**:CLI 一样的可观察性
- **不阻塞 UI**:长任务(sweep)走 `QThread`,完成后 `signal result_ready`

### 4.7 复用 `parser/writer/model/diff/sweep` 详解

| GUI 功能 | 复用哪个 core API | 调用频率 |
|---|---|---|
| **打开文件** | `parse_file(path) → InpFile` | 每次 |
| **保存** | `write(inp, path)` | 每次 |
| **树形展示** | `inp.block_list` / `b.statements` / `s.values_raw` | 实时 |
| **改字段** | `b.set(key, val)` / `b.append(key, val)` | 每次 |
| **Diff 对比** | `diff(inp_a, inp_b) → DiffReport` | 每次 |
| **Sweep 配置** | `CaseSweep.from_dict(cfg) / from_yaml / from_csv` | 每次 |
| **Sweep 执行** | `generate(cs) → SweepReport` | 每次 |
| **检测方程** | `detect_equations(inp) → EquationSystemReport` | 每次 |
| **应用 preset** | `TurbulenceKOmegaPreset.apply(inp)` / `TwoTemperaturePreset.apply` / `SpeciesPreset.apply` | 每次 |

## 6. 风险评估

| 等级 | 风险 | 缓解 |
|------|------|------|
| **HIGH** | **Qt 5.15 EOL 但仍依赖**:Qt 公司 2020-12 不再为 Qt 5.15 出 Win7 修复;PySide2 5.15.2.1 是 2021 最后版,可能有 Win7 未覆盖 bug | 选 **PySide2 5.15.2.1**(不是更早也不是 Qt 6);阶段 6 必须在真 Win7 跑全流程;若发现严重 bug,回退 wxPython(见 §6.1) |
| **HIGH** | **PyInstaller hook 漏 PySide2 子模块**:PySide2 用 `shiboken2` 动态加载,某些 platform plugin 可能漏 | 阶段 6 用 `--collect-all PySide2` 测试,或显式列 `hiddenimports`;参考 PyInstaller 6.16 + PySide2 已知 issue |
| **HIGH** | **Win7 字体/DPI 兼容**:Win7 默认 DPI 100%,但用户可能改 125%/150%;SimSun 字体可能缺失(Win7 简体默认有,繁体/英文版无)| 阶段 6 Win7 烟雾测试必做;QSS 用 Qt 自带字体栈,避免硬编码 SimSun;i18n 走 .ts 资源,font fallback 优雅降级 |
| **MEDIUM** | **打包体积膨胀**:PySide2 + Qt5 plugins 可能 50-80 MB(对比 CLI 的 5-10 MB)| UPX 压缩测试(已知 PyInstaller 杀软误报问题);`--exclude-module` 掉不用的 Qt 模块(如 QtNetwork / QtSql);`onedir` 模式减小单文件(Win7 解压 0.5s 起步)|
| **MEDIUM** | **Win7 无 WebView2 / Edge Chromium**:若有 web 内嵌需求会卡;但 v0.10 不需要,本计划明确不嵌 web | 不引入 `QtWebEngine`(即使 PySide2 自带,Win7 不支持);FastAPI 仍走 `inp-tool-api` 单独启动 |
| **MEDIUM** | **REPL/wizard 共享逻辑不重做**:v0.7 wizard / v0.9.1 检测,GUI 复用即可,不要重写 | 阶段 4 检测面板直接调 `detect_equations()`,不复刻逻辑 |
| **MEDIUM** | **CFD++ 2T 字段名 `vibtem` 是约定**:v0.9.1 引入,GUI 也要带这约定 | 阶段 4 在 DetectPanel 显示"字段名约定"提示;若 v0.9.1 改字段名,GUI 同步 |
| **LOW** | **conda `cfdchanger` Py3.8 上 PySide2 wheel 兼容性**:PySide2 官方 wheel 覆盖 3.8-3.11 Win x86/x64 | 阶段 1 跑 `pip install PySide2==5.15.2.1` 验证;若 wheel 不可用,改 pip mirror(清华 / 阿里) |
| **LOW** | **测试覆盖率难达 80%**:GUI 代码大量 signal-slot,mock 成本高 | 业务逻辑放 controller(纯 Python),widget 只做展示 → controller 易测;widget 测试降到 60% 也接受 |
| **LOW** | **Win7 虚拟机昂贵**:用户自测可能没有 Win7 物理机 | 阶段 6 写 Win7 自测 checklist;用户在自己的 Win7 机器跑(若需);或 docker `dockurr/windows-7` 镜像(社区方案)|
| **LOW** | **commit history 整洁**:6 阶段 ≈ 6+ PR,每个 PR 都要有 conventional commit | 沿用现有 `feat:` / `refactor:` / `docs:` 规范 |

### 6.1 Plan B:若 PySide2 在 Win7 上踩坑

回退方案:**wxPython Phoenix 4.2.3**(wxWidgets 3.2 绑定,Win7 原生外观,体积 +20-30 MB)。

切换成本:
- `widgets/` 目录整体重写(QWidget → wxWindow 父子类,QTreeWidget → wxTreeCtrl,QFormLayout → wxFlexSizer)
- 业务逻辑(controllers/)零修改
- PyInstaller hook 切换
- 估 +1 周工作量

**触发条件**:阶段 6 在 Win7 上发现 PySide2 严重 bug 且 1 周内无解。

## 8. 决策点 / 待用户确认

下列决策点需用户拍板后再进入实施:

### 8.1 框架选型(必须)

| 选项 | 优点 | 缺点 |
|---|---|---|
| **A. PySide2 5.15.2.1**(推荐)| 控件强 / LGPL / 打包成熟 | 体积大(50-80 MB) / Qt 5.15 EOL |
| **B. wxPython Phoenix 4.2.3** | 体积小(20-30 MB) / 原生外观 | API 偏 C++ 风格 |
| **C. Tkinter** | 零依赖 / 体积 0 MB | 朴素 / 树形要手写 |
| **D. PySimpleGUI** | API 极简 | 控件太简单,不适合多步骤向导 |

→ **建议 A**(已写计划),但若用户在意打包体积或 Qt 风险,可改 B。

### 8.2 是否分阶段交付(可选)

- **方案 1**:6 阶段一次到位(v0.10.0),4 周
- **方案 2**:MVP 优先(阶段 1+2+3 = v0.9.2 补丁),2 周交付核心编辑能力;Sweep/Detect/打包后续 v0.10.0

→ **建议方案 1**(Win7 部署要一次到位,半成品没用)。**但若用户希望 v0.9.2 就有 GUI,改方案 2**。

### 8.3 是否 fork 现有 web GUI(可选)

项目已有 `inp_tool/web/index.html` + FastAPI 后端。理论上可以做"桌面端 web 包装":

- **方案 E**:用 Qt WebEngine 嵌本地 FastAPI 页面
- **方案 F**:PySide2 纯 native(本计划方案)

→ **强烈建议 F**,理由:
- Qt WebEngine **Win7 不支持**(Chromium 内核太新,放弃)
- 现有 web 是开发原型,不是为生产设计
- 改写成本与从零写 PySide2 相当

**但若用户希望保留 web 前端为 GUI 雏形**,可以先把 web 重构(阶段 1+2 不动 PySide2,改 web 端 v0.9.2 出 v0.9.3 再 v0.10 PySide2 包装)。这条路要 8+ 周,不推荐。
